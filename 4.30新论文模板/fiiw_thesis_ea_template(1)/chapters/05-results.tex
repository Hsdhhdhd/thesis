\chapter{Results}\label{ch:results}

This chapter reports the outcomes corresponding to the methods in Chapter~\ref{ch:methods}. The results are organised by the four thesis objectives: prototype implementation, static-sitting versus pelvic-rotation detection, Shimmer/XIAO signal comparability, and compact embedded deployment feasibility.

\section{Objective 1: Prototype Implementation Outcome}\label{sec:results_prototype_outcome}

The final prototype implemented the complete sensing-to-cue pipeline defined in Chapter~\ref{ch:methods}. It combined the XIAO nRF54L15 Sense IMU node, BLE streaming, Android field logging with row-level labels, PC-based monitoring, window-level feature extraction, compact KNN inference, a 10~min inactivity timer, a DRV2605L haptic driver, and a one-second coin-motor vibration cue.

Figure~\ref{fig:prototype_views} shows the assembled prototype and its internal battery arrangement. The black carrier plate holds the visible electronics, while the orange enclosure houses the internal battery and provides the flat surface used for body attachment. The assembled prototype therefore met the Objective~1 requirement of integrating sensing, wireless data collection, annotation support, embedded decision logic, and local cue delivery in one low-cost wearable research unit. The field wearing setup used during data collection is described in Section~\ref{sec:labelled_dataset}.

The prototype was worn close to the sacral/pelvic region using a sterile island dressing and double-sided tape. The Android application and PC receiver both used the same BLE packet format, so bench inspection, field logging, and later embedded deployment were based on the same six-axis IMU stream.

The Android front end is shown in Figure~\ref{fig:android_frontend_outcome}. The discovery screen contained a device list showing BLE names and addresses, plus \texttt{Scan} and \texttt{Connect} buttons. The label screen displayed each connected wearable as a card with name, address, \texttt{Disconnect}, ready-to-label status, four label buttons (\texttt{standing}, \texttt{move}, \texttt{nomove}, \texttt{standupanddown}), and \texttt{Stop Label}. The telemetry screen used one card per wearable and showed packet rate, acceleration \(x/y/z\) in \si{\meter\per\second\squared}, and gyroscope \(x/y/z\) in \si{\degree\per\second}. This allowed the researcher to select nearby wearables, label each device separately, and check live streaming during field collection.

\section{Objective 2: Pelvic Rotation Detection}\label{sec:results_detection}

The retained classifier dataset consisted of 98 clean labelled segment files: 66 static-sitting segments and 32 pelvic-rotation segments. The held-out participant test set was imbalanced, with static sitting outnumbering pelvic rotation 21:1.

Table~\ref{tab:ch6_model_results} reports the held-out performance of all five classifiers on participant 5. KNN achieved the highest pelvic-rotation F1-score at 0.957.

Figure~\ref{fig:shimmer_xiao_comparison} shows the aligned time-series comparison for the co-located recording. \subsection{Pilot Field-Worn Trend Agreement}

The pilot protocol was intended to collect paired Shimmer/XIAO field recordings for all five participants, but only two produced usable paired recordings; the other three paired reference recordings failed because the Shimmer data were not successfully recorded. These failed reference logs were not used for the field-worn Shimmer comparison, but the corresponding XIAO recordings were still used where clean labelled seated segments were available for the classifier dataset. Table~\ref{tab:field_recording_summary} summarises the two usable paired field recordings.

The XIAO logs contained 57 gaps longer than 1~s for P1 and 4 for P2. The global rates above include this dead time; the labelled clean segments used for the classifier run at a higher effective active-recording rate. This distinction matters because the field-worn Shimmer comparison uses long overlapping records, whereas the classifier dataset uses manually selected clean seated segments.

Acceleration motion magnitude correlated at 0.918 for P1 and 0.855 for P2, while gyroscope magnitude correlated at 0.908 for P1 and 0.864 for P2. After lag correction, maximum cross-correlations reached 0.915--0.981. The estimated best lag was 1.2--1.3~s (Table~\ref{tab:field_trend_results}). \section{Objective 4: Compact KNN and On-Device Timing}\label{sec:results_embedded_deployment}

The compact prototype-based KNN (12 features, 128 KMeans prototypes), described in Section~\ref{sec:compact_knn}, achieved accuracy 0.996, balanced accuracy 0.963, precision 0.994, recall 0.927, and an F1-score of 0.959 on the held-out participant. The full baseline used 90 features and 9{,}391 reference windows, whereas the compact model used 12 features and 128 prototypes (Figure~\ref{fig:full_vs_compact_knn}).

\begin{figure}[H]
    \centering
    \includegraphics[width=0.92\linewidth]{ch6_full_vs_compact_knn_comparison.png}
    \caption{Comparison between the offline full KNN (90 features, 9{,}391 reference windows) and the compact prototype-based KNN (12 features, 128 KMeans prototypes) used for embedded deployment.}
    \label{fig:full_vs_compact_knn}
\end{figure}

Table~\ref{tab:knn_timing} summarises per-window inference timing measured across 24 consecutive non-overlapping 2~s windows on the XIAO nRF54L15 Sense. The timing measurement includes the compact feature extraction path and the KNN distance computation for the stored prototypes.

\begin{table}[H]
    \centering
    \footnotesize
    \caption{On-device execution time of the compact KNN.}
    \label{tab:knn_timing}
    \begin{tabularx}{\linewidth}{lXX}
        \toprule
        Stage & Mean & Range \\
        \midrule
        KNN inference only           & 2.70~ms & 2.67--2.73~ms \\
        Features + KNN total         & 2.73~ms & 2.70--2.76~ms \\
        \bottomrule
    \end{tabularx}

The 2.73~ms total is below 0.15\% of the 2{,}000~ms window period. The deployed firmware uses 56.3~KB of flash and 9.3~KB of RAM, leaving headroom within the target's available flash and RAM budget for the Zephyr RTOS, BLE stack, and future extensions. These values describe compute and memory feasibility.
